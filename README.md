## What Was Built

### Backend — Laravel 13 REST API
- **4 endpoints:** `GET /api/todos`, `POST /api/todos`, `PUT /api/todos/{id}`, `DELETE /api/todos/{id}`
- **SQLite** database via Laravel migrations and Eloquent ORM
- **DDD-style layering:** `Controller → Service → Eloquent Model` — no business logic in controllers
- **Filter support:** `GET /api/todos?filter=active|completed` using Laravel's `when()` query builder
- **Validation** on all mutating endpoints with proper `422` responses; `404` for missing resources; `500` for model-event failures
- **33 tests, 95 assertions** — all passing

### Frontend — Nuxt 3 + Vuex
- **Nuxt server proxy layer** — the browser never talks to Laravel directly; all requests go through `server/api/todos/`
- **Vuex** store with namespaced `todos` module: `state`, `mutations`, `getters`, `actions` *(For a greenfield Nuxt 3 project I would recommend [**Pinia**](https://pinia.vuejs.org/) — it is the officially recommended store for Vue 3 and has a simpler API, better TS inference, and active maintenance. Vuex was used here only because the assessment required it.)*
- **Backend filtering** — selecting All / Active / Completed triggers a re-fetch via the Laravel filter endpoint (`setFilter` action)
- **4 components:** `TodoForm`, `TodoItem`, `TodoList`, `TodoFilter`
- **Light / dark theme** toggle with hero background image swap; Josefin Sans font; responsive at 375px and 1440px
- **62 tests** — all passing

## Why a Nuxt server proxy instead of calling the API directly

The frontend never talks to Laravel from the browser. Every request goes `Component → Vuex action → utils/todos.ts → Nuxt server route → Laravel`. That extra hop pays for itself:

- **The backend URL is server-only.** It lives in `runtimeConfig` and never reaches the bundle, so the API host can change without redeploying the client and is not advertised to anyone reading the page source.
- **CORS becomes a non-issue.** The browser only ever calls same-origin `/api/todos`, so no preflight requests, no CORS headers to maintain, and no environment-specific origin lists.
- **Single place to add cross-cutting concerns.** Auth tokens, request signing, header injection, rate limiting, response shaping, retries, and logging all live in one server-side layer instead of being sprinkled across components.
- **Browser surface stays minimal.** Secrets, internal hostnames, and infrastructure details are never exposed to the client — only the proxy sees them.
- **Backend can move freely.** Switching the Laravel host, putting it behind a VPN, or replacing it with a different service is a one-line change in the proxy. No client code changes.
- **SSR-friendly.** The same `$fetch('/api/todos')` call works during server rendering and on the client, so hydration and direct navigation behave identically.
- **Cleaner tests.** Components and the store mock a stable internal contract (`/api/todos`) instead of being coupled to the Laravel host.

## Why I write skill files for the AI the way I do

Inside [`.claude/commands/`](.claude/commands) there are two skill files — [`new-endpoint.md`](.claude/commands/new-endpoint.md) and [`new-component.md`](.claude/commands/new-component.md) — that I authored by hand. They encode, step by step, the exact way I personally build a REST endpoint and a frontend component: the order of operations, the questions I ask before writing code, the layers I separate, the anti-patterns I refuse to ship, and the commit cadence I follow. Writing the AI's instructions this way pays off in concrete ways:

- **Determinism over drift.** A numbered, ordered workflow ("Step 0 — Analyze & Ask", "Step 1 — Tests first", …) removes the AI's freedom to take shortcuts. Every endpoint and every component goes through the same pipeline, so the codebase looks like one person wrote it.
- **TDD is enforced, not requested.** Tests come before implementation in Step 1, every time. The AI cannot "forget" to write tests because there is no implementation step to jump to first.
- **Architecture is encoded, not hoped for.** The skill files name the exact directories (`Services/{Domain}/`, `server/api/`), the exact file conventions (`index.get.ts`, `[id].put.ts`), and the exact responsibilities of each layer. The AI does not have to guess where a service belongs — it is told.
- **Anti-patterns are explicit.** Lists like "no business logic in controllers", "no raw `fetch()` in components", "never commit `SET_FILTER` directly from a component" close off the failure modes I care about *before* the AI can produce them.
- **Domain rules live in the skill, not in chat.** Things like "`status` defaults to `false`", "`GET /todos` returns newest first", "filtering happens on the backend" are written down once and re-applied every time the skill runs, so I never have to re-explain them.
- **Commit hygiene is built in.** "One commit per step, never batch" produces a git history that is reviewable as a sequence of small, intent-named changes — exactly what I would want from a human teammate.
- **Mandatory analysis step.** Step 0 forces the AI to ask clarifying questions before writing code. That single rule eliminates most of the "you built the wrong thing" loops.
- **The skill file is the spec.** When the project grows, the next change starts by editing the skill, not by re-explaining requirements in chat. The instructions become a living architectural document that future contributors (human or AI) can read and follow.

In short: the skills make the AI behave like a disciplined senior engineer who already knows this codebase, instead of a fast-but-generic generator. The cost is paid once, up front, in writing the skill carefully — and recovered every single time a new feature is built.

## Assessment

The objective of this challenge is to build a full-stack todo application utilizing Laravel as the back-end API and Nuxt for the front-end.

## Brief

In this exercise, you will create a simple, yet functional, Todo application. Users will be able to add, update, and delete tasks. They will also have the ability to filter tasks based on their completion status. The design will be up to your discretion.

**Server Side**

1. Set up a new Laravel project.
2. Implement the following API endpoints:
    - `GET /todos` - Fetch all todos.
    - `POST /todos` - Create a new todo. This should accept a JSON body with the `content` of the task and its `status`.
    - `PUT /todos/{id}` - Update a todo. This should accept a JSON body with the new `content` and/or `status`.
    - `DELETE /todos/{id}` - Delete a todo.
3. Each todo item should have the following fields:
    - `id` (integer)
    - `content` (string)
    - `status` (boolean)
4. Implement a SQLite database using Laravel's database migrations and Eloquent ORM to store the todos.
5. Write tests to ensure the API endpoints are working as expected.

**Client Side**

1. Set up a new Nuxt project.
2. Create the following pages:
    - Home page (`/`) - Display all todos, with the ability to filter by their status (all, active, completed).
3. On the home page, implement the following functionality:
    - Add a new task.
    - Mark a task as completed.
    - Delete a task.
    - Filter tasks based on their status.
4. Use Vuex for state management.
5. Write tests to ensure the UI behaves as expected.

### Evaluation Criteria

- We're looking for you to produce working code, with enough room to demonstrate how to structure components in a small program
- Completeness: did you complete the features?
- Correctness: does the functionality act in sensible, thought-out ways?
- Maintainability: is it written in a clean, maintainable way?
- Testing: is the system adequately tested?

Please organize, design, test, and document your code as if it were
going into production - then push your changes to the master branch.
